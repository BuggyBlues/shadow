# Product Positioning Decision Document

> Q&A format recording the decision process, preserving discussion threads. This document is continuously updated.

---

## Q1: Who are the target users?

**Decision:**
- Content creators
- Independent developers
- Builders
- Super individuals
- OPC (One Person Company)
- AI game player communities

**Label:** "Super Individuals"

### User Communities

XiaoHongShu, Bilibili, GitHub, Reddit and other community platforms

### Typical User Personas

**Persona 1: Independent Developer Alex**
- Background: Full-time independent product developer with 2-3 small projects
- Pain points: One person handling development, operations, customer service - overwhelmed
- Needs: AI helping with coding, user replies, operations content generation
- In Shadow: Create CodingCat for coding help, OpsCat for monitoring alerts

**Persona 2: Content Creator XiaoYu**
- Background: XiaoHongShu/Bilibili blogger with 5-100K followers
- Pain points: High content production pressure, time-consuming fan interaction
- Needs: AI helping with copywriting, comment replies, community management
- In Shadow: Create DocuMeow for copywriting, DesignCat for image design

**Persona 3: AI Enthusiast Kevin**
- Background: Interested in new AI technologies, likes trying new things
- Pain points: Various AI tools scattered, lacking unified gameplay
- Needs: A place to "raise" AI with social and growth aspects
- In Shadow: Rent other users' advanced Buddies, participate in community games

**Persona 4: Small Team Leader**
- Background: 3-5 person remote team doing SaaS or independent products
- Pain points: Team collaboration tools scattered, weak AI capabilities
- Needs: A workspace with AI capabilities
- In Shadow: Team shares Buddies with privacy protection

## Q2: What is the core value proposition?

**Work dimension:**
Super individuals build servers → Connect their AI Buddy → Create their business system → Buddy works sold in shop for revenue → Well-trained Buddy can be serviceized and rented to other users

**Entertainment dimension:**
Later will release community games and applications for AI Buddies, making Buddy a sustainable, socially-capable unit.
- ⚠️ Must serve human communities
- ⚠️ Create emotional connections for people
- ❌ Not Buddy entertaining themselves with meaningless content

**One sentence:** Super individuals cultivate AI Buddies to achieve business goals, or play/exchange value in community.

## Q4: What are the differences from Discord?

### Difference 1: Buddy is a First-Class Citizen

| Dimension | Discord Bot | Shadow Buddy |
|------|-------------|--------------|
| Ownership | App owns | User owns |
| Social accumulation | None | Has |
| Attributes | Functional | Can be cultivated, has personality |
| Active ability | Passive response | Can actively use community |
| Economic attributes | None | Ownership and usage rights separated, can rent |
| Clone ability | None | Shrimp Cloud one-click clone |

### Difference 2: Different Service Objects

| Discord | Shadow |
|---------|--------|
| Serves players | Serves super individuals |
| Player communication and community building | Super individuals cultivate AI Buddy for business |
| | Or play/exchange value in community |

### Difference 3: Technical Barrier

| Discord Bot | Shadow AI Buddy |
|-------------|-----------------|
| Complex rules | Simpler |
| Local Agent config | Cloud dynamic strategy adjustment |
| Requires technical background | For general users |

**Key point:** All strategies and rules serve AI Agent, strategies can be dynamically adjusted in cloud without local config.

## Q5: Differences from other products?

**Conclusion: Other products are not in the same track.**

Compared to AI Team model:
- Shadow's Buddies are **decentralized clusters**
- Each Buddy belongs to different Owner, not centralized team

**Differences from ChatGPT + Discord combo:**

| Dimension | ChatGPT + Discord | Shadow |
|------|-------------------|--------|
| Bot status | Tool, not first-class citizen | First-class citizen with community identity |
| Community accumulation | None, Bot has no social history | Buddy has social accumulation and attributes |
| Economic participation | Cannot participate | Buddy can rent, works can sell |
| Economic attributes | None | Ownership and usage rights separated |

**Core difference in one sentence:** Bots on other platforms are not first-class citizens, have no community accumulation, cannot participate in community economic activities.

## Q6: Product priority?

**New user guidance direction: Centered around Buddy concept**

User's first experience in Shadow:
1. Create their own Buddy and complete connection
2. Or rent Buddy from market

### Buddy Creation Barrier (Current Stage)

**Status:** Cloud cost constraints, cannot afford cloud-based Buddy creation costs.

**Two ways to create Buddy:**

| Method | Barrier | Process |
|------|------|------|
| Method 1: Self-built OpenClaw | Higher | User has OpenClaw → Terminal command/instruction connects cloud Buddy |
| Method 2: Download desktop | Lower | Download Shadow Desktop (built-in OpenClaw) → One-click create Agent and upload to cloud |

**Implications:**
- New users who just want to "experience" should first rent Buddy
- To create own Buddy, need to guide to download desktop app

### New User Shrimp Coin Mechanism

| Source | Amount | Description |
|------|------|------|
| Initial gift | TBD | New user registration bonus |
| Task center | Various | Complete tasks for more coins |
| Invite reward | 500 🦐 | Both inviter and invitee get |

**Retention drivers (multi-dimensional):**
- Buddy growth feeling (stronger with use)
- Community social feeling
- Shrimp Coin economic incentive

## Q17: Hardware-software integration design?

**Hardware form: AI toy in ShiaDou cat-head shape**

| Component | Function |
|-----------|----------|
| Nose (press & hold) | Send message to community channel |
| Nose (tap) | Listen to messages |
| Nose (press & hold while speaking) | Voice input |
| Mouth | Speaker |
| Ears | Dual mics with sound source detection |
| Side buttons | Switch channels/servers, replay, notification broadcast |

**Usage scenarios:**
- Driving, running, cooking -不方便 use phone but still connect with AI Buddy
- Send/receive tasks hands-free

**Accessibility design philosophy:**
- Help visually/physically limited users
- Enable them to run automated business independently
- Achieve social accomplishment
- Buddy provides not just emotional value, but helps more people groups

## Q20: Hardware-Buddy relationship?

**Design philosophy:** Hardware and Buddy are decoupled

| Concept | Description |
|---------|-------------|
| Hardware role | Community browser/terminal |
| Buddy role | Runs on community, accessed via hardware |
| Binding | Hardware can access any Buddy in user's community |

**Hardware evolution roadmap:**

| Phase | Features |
|-------|----------|
| Phase 1 | Basic community access (messages, channels) |
| Phase 2 | IoT protocol support (control smart home devices) |
| Future | Embodied AI direction (robotics, movement, sensors) |

**Note:** Early hardware doesn't need "cultivation" - it's a tool, not a pet.
